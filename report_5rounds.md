# 面试评估报告 (Hire Packet)

## 1. Executive Summary
该候选人具备极佳的结构化思维与高并发系统建模能力，在流量过滤与自适应限流领域展现了深厚的理论功底（如 Little's Law 的应用）。然而，作为 Staff/Principal 级别候选人，在应对“绝对强一致性”这一核心约束时，其方案在 Redis 节点宕机回滚、原子性断裂（DECR 与凭证写入）等极端故障场景下存在明显的逻辑盲区，未能真正打通从缓存到数据库的闭环一致性链路。

## 3. Dimensional Evaluation

- **Discovery & Metrics: Solid**
    候选人能够精准捕捉“锁定座位”这一核心状态机转换，并给出了具备实操性的对账公式（`DB已售 + Redis库存 = 总量`）。不足之处在于对系统可观测性的定义流于表面，未能给出针对 10M 级并发冲击下的 P99 响应时间目标或库存审计的具体一致性 SLIs。

- **Math & Probabilistic: Solid**
    在 10w 级库存的内存占用及网关带宽负载计算上，逻辑严密，符合物理常识。遗憾的是，候选人未能将 1000w 峰值流量进一步解构为各核心微服务组件的 QPS 负载分布，缺乏对系统全局压力的精细化量化。

- **Logical Consistency: Marginal**
    这是该候选人的主要失分项。虽然其理解唯一索引是防超卖的最后防线，并提出了基于 ZSet Sharding 的对账优化，但在 Staff 级别必须关注的容错细节上表现欠佳：方案过度依赖 Redis 异步复制，未能识别 Master 宕机导致的数据回滚（Rollback）风险；同时，方案中 Redis 扣减与凭证写入缺乏原子性保护，在应用崩溃时会造成“库存黑洞”。对于“绝对强一致性”的理解仍停留在快乐路径（Happy Path），对分布式异常处理的思考深度不足。

- **Tradeoffs & Rationale: Solid**
    在 Redis Lua 与 MySQL 事务的选择上，候选人展示了清晰的选型逻辑，能够基于性能与复杂度的平衡给出合理的架构决策。

- **Ilities & Execution: Outstanding**
    候选人在系统弹性设计上表现出色。其提出的“流量漏斗模型”层级分明，尤其是利用利特尔法则（Little's Law）构建的自适应限流闭环，以及基于梯度算法的流量反馈机制，体现了 Staff 级别应有的工程前瞻性。

## 4. Interview Notes

### 核心设计路径：
1. **流量接入层（主动提出）：** 候选人主动构建了从 CDN、WAF 到虚拟排队机的多级过滤体系。特别是在排队机环节，能够利用梯度算法进行自适应限流，这在处理突发 10M 流量时是非常成熟的考虑。
2. **库存处理层（主动提出）：** 提出了 Redis Lua 扣减 + ZSet 凭证的组合方案，并意识到 ZSet 扫描的性能瓶颈，主动给出了 Sharding 方案进行优化。
3. **一致性保障（追问后补齐）：** 初始方案倾向于异步最终一致性（Redis -> MQ -> DB）。在被反复追问如何实现“绝对强一致性”时，候选人补齐了数据库唯一索引作为物理终点，并提出了“强制熔断”作为极端情况下的保护手段。

### 表现盲区：
1. **原子性断裂问题（始终未答上来）：** 在深度讨论分布式环境下 Redis 扣减（DECR）与本地凭证写入的原子性时，候选人始终没能给出一个严谨的补偿或同步写入机制，无法彻底解决应用层崩溃导致的库存丢失问题。
2. **底层背压机制（回避）：** 当问及底层服务抖动如何反向调节前端流入速度（Backpressure）时，候选人避开了具体算法（如 AIMD）的实现细节，反映出其对底层通信控制逻辑的掌握略逊于高层架构设计。
3. **容灾边界（未识别）：** 候选人默认了 Redis 及其主从架构的高可用性，但对于 Staff 级别必须考虑的“主从切换导致的数据丢失”以及由此引发的超卖风险，未能给出主动的防御对策。
