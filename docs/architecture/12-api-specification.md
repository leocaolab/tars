# Doc 12 — API Specification (Rust / HTTP / gRPC / Python / TypeScript / WASM)

> Scope: defines all consumer-facing API surfaces — the in-process Rust trait, HTTP REST + SSE, gRPC, Python FFI binding, TypeScript binding, and the optional WASM build.
>
> Context: every API is a different projection of the same Doc 04 `Runtime` trait. Rust is the source of truth; the other bindings are either generated by tooling or hand-maintained as thin shells.

---

## 1. Design Goals

| Goal | Description |
|---|---|
| **Single source of truth** | The Rust trait is truth; every other API is projected from it. schema / proto / pyi / d.ts are either generated or kept aligned |
| **Optional protocol cost** | In-process embedding uses FFI (PyO3/napi-rs); distributed deployments use HTTP/gRPC. The customer chooses |
| **Streaming is mandatory** | Every API supporting long tasks must expose an event stream, not just request-response |
| **Type safety** | Strongly typed languages (Rust/TS) get full types; Python via .pyi stubs |
| **Version compatibility** | API evolution has a contract; backward compatibility holds between major bumps |
| **Unified errors** | The error model is semantically equivalent across all languages, not just translated strings |
| **Unified auth** | Token / OIDC / mTLS behave consistently across every protocol |
| **Discoverable, testable** | OpenAPI / protobuf / .pyi / d.ts are auto-generated; interfaces are consumable by tooling |

**Anti-goals**:
- No unmaintained bindings for the sake of "covering every language" — priority follows actual demand
- No language-specific binding is allowed to drift from core semantics — every binding goes through the same conformance test set
- No business logic in the binding layer — bindings are thin shells, logic lives in the Rust core
- No exposing internal Rust data structures (e.g. `Arc<Trajectory>`) externally — expose via ID + query

## 3. Native Rust API

This is the **contract source** for every other API. See Doc 04 §12 for details; this section is a brief recap:

```rust
#[async_trait]
pub trait Runtime: Send + Sync {
    async fn submit(&self, spec: TaskSpec, principal: Principal) 
        -> Result<TaskHandle, RuntimeError>;
    
    fn subscribe(&self, task: TaskId) 
        -> BoxStream<'static, TrajectoryEvent>;
    
    async fn cancel(&self, task: TaskId) -> Result<(), RuntimeError>;
    async fn suspend(&self, task: TaskId) -> Result<(), RuntimeError>;
    async fn resume(&self, task: TaskId, trigger: ResumeTrigger) -> Result<(), RuntimeError>;
    
    async fn query(&self, filter: TaskFilter) 
        -> Result<Vec<TaskSnapshot>, RuntimeError>;
}
```

**Integration example** (Rust application embedding):

```rust
use tars_runtime::{Runtime, RuntimeConfig, TaskSpec};
use std::sync::Arc;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let config = RuntimeConfig::from_file("./config.toml")?;
    let runtime: Arc<dyn Runtime> = tars_runtime::build(config).await?;
    
    let task = runtime.submit(
        TaskSpec::skill("code-review")
            .arg("repo", "github.com/me/myproject")
            .arg("pr", 42),
        Principal::current_user()?,
    ).await?;
    
    let mut events = runtime.subscribe(task.id);
    while let Some(event) = events.next().await {
        println!("{:?}", event);
    }
    
    Ok(())
}
```

## 4. HTTP REST + SSE

### 4.1 Why REST + SSE rather than pure gRPC

- **Low client barrier** — every language has an HTTP client
- **Browser friendly** — SSE has native streaming support
- **Debugging friendly** — curl / Postman / browser devtools
- **Cost**: serialization is ~30% less efficient than gRPC — but LLM calls take seconds anyway, that 1ms HTTP overhead does not matter

### 4.2 OpenAPI Spec

The full spec lives at `api/openapi.yaml`, derived from Rust code (via `utoipa`):

```rust
pub struct ApiDoc;
```

**Key constraints**:
- `Content-Type: text/event-stream` triggers browser EventSource handling
- `X-Accel-Buffering: no` disables nginx buffering
- Each event is `event:` + `data:` two lines + blank line
- Heartbeat event `event: ping` every 15s, prevents proxy idle disconnects

## 5. gRPC

### 5.1 When to use gRPC

- Internal communication between Rust microservices
- High throughput scenarios (> 1000 RPS)
- Bidirectional streaming (e.g. sub-agent runtimes talking to each other, Doc 05 §6.4)

### 5.2 protobuf Definition

`api/proto/runtime.proto`:

```protobuf
syntax = "proto3";
package tars.runtime.v1;

import "google/protobuf/timestamp.proto";
import "google/protobuf/struct.proto";

service Runtime {
  rpc Submit(SubmitRequest) returns (SubmitResponse);
  rpc Subscribe(SubscribeRequest) returns (stream TrajectoryEvent);
  rpc Cancel(CancelRequest) returns (CancelResponse);
  rpc Suspend(SuspendRequest) returns (SuspendResponse);
  rpc Resume(ResumeRequest) returns (ResumeResponse);
  rpc Query(QueryRequest) returns (QueryResponse);
}

message SubmitRequest {
  string skill_or_blueprint = 1;
  google.protobuf.Struct args = 2;
  TaskBudget budget = 3;
  optional string session_id = 4;
  Principal principal = 5;
}

message TrajectoryEvent {
  string task_id = 1;
  google.protobuf.Timestamp timestamp = 2;
  oneof event {
    TaskStarted task_started = 10;
    AgentInvoked agent_invoked = 11;
    AgentCompleted agent_completed = 12;
    PartialArtifact partial_artifact = 13;
    Completed completed = 14;
    Failed failed = 15;
    // ... see Doc 04 §12
  }
}

// Error details
message ApiError {
  string code = 1;
  string message = 2;
  string trace_id = 3;
  google.protobuf.Struct details = 4;
}
```

`tonic-build` generates the Rust + client SDK.

### 5.3 Bidi Stream

Sub-Agent runtime scenario: a parent runtime invokes a child runtime; cancel / suspend signals must flow bidirectionally:

```protobuf
service Runtime {
  rpc StreamingTask(stream TaskCommand) returns (stream TrajectoryEvent);
}

message TaskCommand {
  oneof command {
    SubmitRequest submit = 1;
    string cancel_task_id = 2;
    SuspendRequest suspend = 3;
    ResumeRequest resume = 4;
  }
}
```

### 5.4 mTLS

Internal gRPC requires mTLS:

```rust

Server::builder()
```

## 6. Python Binding (PyO3)

Projects Doc 06's **config + role-resolution** model (global config, loaded once →
free-function role resolvers → per-call context → provider / pipeline). After the
scope-facade simplification there is **no** per-workspace handle object: `tars-py`
exposes module-level functions (`tars.init`, `tars.provider(role)`,
`tars.pipeline(role)`, `tars.context(...)`) over the process-global `Config` +
`ProviderRegistry`. There is also **no** task-submission surface
(`submit(TaskSpec, Principal)` / `subscribe`) — that server/task shape is gone.

### 6.1 Design Choice

Two paths:

| Option | Pros | Cons |
|---|---|---|
| **A. PyO3 direct binding to Rust** | In-process, extremely low latency; can pass native types | Compilation complexity (per-platform wheels); Rust upgrades require rebuilding the Python package |
| **B. HTTP client + Pydantic types** | Fully cross-process, usable from any Python; the HTTP service can be upgraded independently | Network latency; serialization on every request |

**Default recommendation: A** (PyO3) for: embedding into Jupyter Notebook, ML pipelines, local script tooling
**Default recommendation: B** for: Python web applications calling a remote TARS cluster

### 6.2 PyO3 Binding Design

**Sync-blocking, not async.** The real `tars-py` binding does **not** use
`pyo3-asyncio` / `future_into_py`. It owns one process-wide multi-thread tokio
runtime (`TOKIO`, `lib.rs:65`) and every call does `py.allow_threads(|| rt.block_on(…))`
(`lib.rs:1840`) — releasing the GIL while it waits so other Python threads keep running.
So the Python surface is **plain blocking methods**, not coroutines; concurrency comes
from GIL-release + the internal multi-thread runtime, not from `asyncio`. This matches
Python's common script/notebook idiom (no event loop to spin up).

The binding reuses the existing `Provider` (layer 1) and `Pipeline` (layer 2)
pyclasses and adds a small set of **module-level functions** over
`tars_handle::init_from_home` / `resolve_role` / `RUN_CONTEXT`. The old
`Workspaces` / `Tars` per-scope classes are **gone** — providers are global (built
once by `init`), and a role resolves against the process-global `Config` +
`ProviderRegistry`.

`tars-py` crate (matches `src/handle.rs`):

```rust
// src/handle.rs — the process-global spine (post scope-facade)
use pyo3::prelude::*;
use tars_config::{Config, resolve_home};
use tars_handle::{resolve_role, resolve_workspace_root, workspace_store_dir, WorkspaceResolution};
use tars_provider::ProviderRegistry;

/// Load the process-global config once and eagerly build the one provider
/// registry. `home` maps to `resolve_home` (--tars_home > $TARS_HOME > ~/.tars)
/// and reads `<home>/config.toml` (+ its `.env`). Idempotent: a second call is a
/// no-op (first load wins). Providers / keys are global — built once, shared.
#[pyfunction]
#[pyo3(signature = (home = None))]
fn init(home: Option<PathBuf>) -> PyResult<()> {
    match tars_handle::init_from_home(home) {
        Ok(()) | Err(tars_handle::InitError::AlreadyInitialized) => Ok(()),
        Err(e) => Err(init_to_py(e)),
    }
}

/// Resolve `role` → a raw `Provider` (layer 1, no middleware) against the
/// global config + registry. Requires `init`.
#[pyfunction]
fn provider(role: String) -> PyResult<Provider> {
    let (id, prov) = resolve_global_role(&role)?;
    Ok(Provider::from_provider(id, prov))
}

/// Resolve `role` → a middleware-wrapped `Pipeline` (the canonical default
/// chain) against the global config + registry. Requires `init`.
#[pyfunction]
fn pipeline(role: String) -> PyResult<Pipeline> {
    let (id, prov) = resolve_global_role(&role)?;
    Ok(Pipeline::from_provider(id, prov, vec![], None))
}

/// Inspect the `[roles]` mapping without building anything.
#[pyfunction]
fn role_provider(role: String) -> PyResult<String> { Ok(resolve_global_role(&role)?.0) }

fn resolve_global_role(role: &str) -> PyResult<(String, Arc<dyn LlmProvider>)> {
    let registry = ProviderRegistry::global().map_err(/* … */)?;
    let cfg = Config::get();
    let (id, prov) = resolve_role(&cfg.roles, &cfg.routing, &registry, role).map_err(handle_to_py)?;
    Ok((id.to_string(), prov))
}

/// Context manager: establishes a `RequestContext` for the calls inside its
/// `with` block. Each `complete()` inside re-scopes `RUN_CONTEXT` from it.
#[pyfunction]
#[pyo3(signature = (session = None, tags = None, tenant = None, trace = None))]
fn context(session: Option<String>, tags: Option<Vec<String>>,
           tenant: Option<String>, trace: Option<String>) -> ContextGuard {
    ContextGuard::new(session, tags, tenant, trace)
}

// Kept path helpers (plain "where does it live", NOT a scope handle):
//   resolve_root(tool, path) -> Option<String>   — canonical workspace root
//   store_dir(tool, root)    -> String           — <root>/.<tool>/tars/
```

The per-workspace store survives only as a **path helper** (`store_dir`) the caller
opens itself — tars no longer bundles roles + store + cancel into a scope object.
Every method that reaches the network follows the `Provider`/`Pipeline` precedent:
`py.allow_threads(|| TOKIO.block_on(async { RUN_CONTEXT.scope(ctx, …).await }))` — the
ctx from the active `context(...)` is re-scoped inside the block, because `task_local`
does not survive the FFI hop. The model is bound on the service, so `complete(model, …)`
takes the concrete model as its first argument (the `ChatRequest` carries none).

### 6.3 Python Usage

The spine — `init` → `with context` → provider / pipeline:

```python
import tars

tars.init()                                        # global config, once (~/.tars, + .env)

with tars.context(session="sess-abc", tags=["dogfood"]):
    # layer 1 — one raw call, no middleware
    resp = tars.provider("critic").complete("deepseek-chat", user="review this")

    # layer 2 — same surface, middleware (cache/retry/telemetry) engaged
    resp = tars.pipeline("critic").complete("deepseek-chat", user="review this")
    print(resp.text, resp.telemetry)               # cache_hit, retry_count, latency, …

# where the per-project store lives, if the caller wants to open one itself:
root = tars.resolve_root("arc", "/repo/myproject") # canonical workspace root, or None
store = tars.store_dir("arc", root)                # <root>/.arc/tars/
```

`tars.provider(role)` / `tars.pipeline(role)` resolve the role against the global
`[roles]` map + registry; `complete(model, …)` takes the concrete provider-side model as
its first argument. The **layer-3 agent-runtime / DAG** surface (a `Plan` of role-based
stages) is **deferred** in the bindings — it lands in a follow-on and is driven from Rust
(`tars_runtime::run_plan`) today, not from Python.

## 7. TypeScript / Node Binding

Mirrors the same Doc 06 spine as Python (global config, once → free-function role
resolvers → per-call context → provider / pipeline), but in the **JS-idiomatic** shape:
napi functions with `complete` returning a **Promise**, so callers `await`. That async
choice is fine and expected here (unlike Python's deliberate sync-blocking); the ctx
boundary law still holds — ctx is passed explicitly and re-scoped per call inside the
binding, never carried across the FFI hop as a `task_local`. Like `tars-py`, `tars-node`
dropped the `Workspaces` / `TarsHandle` per-scope classes: `init` / `provider` /
`pipeline` are free functions over the process-global config + registry. No
`submit(TaskSpec, Principal)` / task model — that is gone.

### 7.1 Three TS Paths

| Path | Use | How tars is reached |
|---|---|---|
| **A. napi-rs direct binding** (`@tars/runtime`) | Node CLI tools / servers | in-process `.node`; TS calls tars directly |
| **B. Tauri command** (desktop app, e.g. concer) | DMG/desktop GUI | **NOT napi** — tars stays entirely in Rust behind Tauri commands; the TS frontend only calls `invoke('cmd', …)` and never sees tars |
| **C. HTTP client** (`@tars/client-http`) | Browser / remote | REST + SSE to a tars server (§7.5) |

> **Desktop apps do not bind tars via napi.** A Tauri app initializes tars once in Rust
> (`tars_handle::init`), keeps whatever per-workspace scope it needs in its own state, and
> exposes its own `#[tauri::command]`s. The webview calls `invoke('cmd', …)`; the global
> registry and `RUN_CONTEXT` live on the Rust side. Only path **A** (Node) actually links
> the napi binding below.

### 7.2 napi-rs binding (`tars-node`)

```rust
// src/handle.rs — free functions over the process-global config + registry
use napi_derive::napi;

/// Global config, once (`home` > $TARS_HOME > ~/.tars; also loads its `.env`).
/// Builds the shared registry eagerly; idempotent (a 2nd call is a no-op).
#[napi]
pub fn init(home: Option<String>) -> Result<(), String> {
    match tars_handle::init_from_home(home.map(PathBuf::from)) {
        Ok(()) | Err(tars_handle::InitError::AlreadyInitialized) => Ok(()),
        Err(e) => Err(init_to_js(e)),
    }
}

/// Resolve `role` → a raw `Provider` (layer 1). Optional `ctx` binds an
/// explicit call context. The concrete model is bound from the role's
/// `default_model` at resolution — the request carries none.
#[napi]
pub fn provider(role: String, ctx: Option<JsContext>) -> Result<Provider, String> {
    let (_id, prov, model) = resolve_global(&role)?;
    Ok(Provider { role, inner: LlmService::of(prov, model), ctx: /* … */ })
}

/// Resolve `role` → a middleware-wrapped `Pipeline` (the canonical default
/// chain). Same shape as `provider`, but the service is `default_chain`.
#[napi]
pub fn pipeline(role: String, ctx: Option<JsContext>) -> Result<Pipeline, String> {
    let (id, prov, model) = resolve_global(&role)?;
    let inner = LlmService::default_chain(prov, model, ChainOpts::new(ProviderId::new(id.clone())));
    Ok(Pipeline::from_service(id, inner, /* ctx */))
}

/// Kept path helpers (plain paths, not a scope handle):
///   resolve_workspace_root(tool, path) -> Option<String>   — canonical root
///   workspace_store_dir(tool, root)    -> String           — <root>/.<tool>/tars/
```

`Provider` / `Pipeline` each carry the bound `LlmService` + a `RequestContext`; their
async `complete(opts)` re-scopes `RUN_CONTEXT` with that ctx at the boundary (Doc 06 §9),
because `task_local` does not cross the FFI hop. The per-workspace store survives only as
a path helper — there is no scope-handle object to `open`/`close`.

### 7.3 TypeScript Usage

The spine — `init` → resolve a role → `complete`:

```typescript
import { init, pipeline, provider } from '@tars/runtime';

async function main() {
  init();                                          // global config, once

  // layer 1 / 2 — raw provider vs middleware-wrapped pipeline. The context
  // (session / tags) is passed explicitly; the model is bound from the role's
  // default_model at resolution.
  const p = pipeline('critic', { session: 'sess-abc', tags: ['dogfood'] });
  const resp = await p.complete({ user: 'review this' });
  console.log(resp.text, resp.telemetry);          // cache_hit, retry_count, latency
}

main().catch(console.error);
```

Errors surface as a typed class hierarchy `TarsError → TarsConfigError /
TarsProviderError / TarsHandleError`, mapped from the Rust typed errors (not a single
stringified message). The **layer-3 agent-runtime / DAG** surface is **deferred** in the
bindings — driven from Rust (`tars_runtime::run_plan`) today, not from Node.

## 8. WASM Binding (Optional)

### 8.1 WASM Subset

The full Runtime **cannot run** in WASM — the following features are unavailable:
- Subprocesses (CLI / MCP) → no syscalls
- Postgres / Redis → no sockets
- Filesystem persistence → IndexedDB / OPFS only

The WASM subset can only do:
- HTTP client calls to a remote TARS server
- Cache key hash computation (local)
- Type validation (schema)
- Token count estimation

### 8.2 Use Cases

- Client-side logic for a browser TARS Web Dashboard
- LLM routing inside Cloudflare Workers / Vercel Edge Functions
- Direct embedding in desktop apps (Tauri)

### 8.3 Implementation

```rust

    token: Option<String>,

            .await?;
        // ... parse response
    }

    pub fn subscribe(&self, task_id: String, on_event: js_sys::Function) {
        let url = format!("{}/api/v1/tasks/{}/events", self.base_url, task_id);
        let es = web_sys::EventSource::new(&url).unwrap();
        // ... bind on_event callback
    }
}
```

## 10. Cross-Language Contract Unification

### 10.1 Single Source of Truth

```
Rust types (src/types.rs)
    │
    ├─ derive(Serialize, Deserialize) → JSON schema
    ├─ derive(ToSchema) → utoipa OpenAPI spec
    ├─ derive(prost::Message) → protobuf (for gRPC)
    ├─ pyo3 binding (manual + macro)
    ├─ napi-rs binding (manual + macro)
    └─ wasm-bindgen (manual)
```

Any field change starts in Rust, then is generated / synced to the other languages.

### 10.2 Conformance Test Suite

Every binding runs the same `conformance_tests/` set:

```
conformance_tests/
  001_submit_and_complete.json     # submit a simple task, verify the full event sequence
  002_cancel_mid_stream.json       # cancel mid-stream, verify cleanup
  003_iam_denied.json              # unauthorized scenario
  004_budget_exceeded.json         # budget exhausted
  005_provider_error.json          # upstream provider error
  006_streaming_concurrent.json    # multiple concurrent subscriptions to the same task
  ...
```

Each test is a declarative spec:

```json
{
  "name": "submit_and_complete",
  "given": {
    "config": "fixtures/config-minimal.toml",
    "mock_provider_responses": [...]
  },
  "when": {
    "actions": [
      { "type": "submit", "spec": {...} },
      { "type": "subscribe", "task_id": "$0.task_id" }
    ]
  },
  "then": {
    "events": [
      { "kind": "task_started" },
      { "kind": "agent_invoked", "agent": "orchestrator" },
      { "kind": "completed" }
    ]
  }
}
```

Each binding implements a conformance runner that runs every test case. New bindings must pass the entire conformance suite before they can ship.

## 11. Versioning

### 11.1 SemVer

- **Major (X.0.0)**: breaking API change (Rust trait signature / HTTP path / proto field removal)
- **Minor (1.X.0)**: new feature, backward compatible (new endpoint / new event type / new optional field)
- **Patch (1.0.X)**: bugfix

### 11.2 Multi-Version Coexistence

HTTP / gRPC coexist via path versioning:

```
/api/v1/tasks
/api/v2/tasks   ← new version, fields may differ
```

Older versions are retained for 12 months after deprecation.

### 11.3 SDK Version vs Server Version

| Scenario | Compatibility requirement |
|---|---|
| SDK v1.x ↔ Server v1.x | Fully compatible |
| SDK v1.x ↔ Server v1.y (y > x) | Server backward compatible, SDK only uses v1.x features |
| SDK v1.x ↔ Server v2.x | `/api/v1/` paths still work, but SDK cannot access new features |
| SDK v2.x ↔ Server v1.x | Rejected, prompts to upgrade server |

The SDK calls `/api/version` at startup to detect:

```http
GET /api/version
{
  "server_version": "1.5.2",
  "supported_api_versions": ["v1"],
  "deprecation_notices": []
}
```

On incompatibility, the SDK throws a clear error rather than letting users debug blindly.

## 12. Unified Authentication

All APIs accept the same set of authentication mechanisms:

| Protocol | Auth header / mechanism |
|---|---|
| HTTP / SSE | `Authorization: Bearer <token>` |
| gRPC | metadata `authorization: Bearer <token>` or mTLS client cert |
| Python (FFI) | Process inherits OS user / explicit Principal::from_token() |
| Node (FFI) | Same as above |
| WASM (browser) | HTTP token (browser cookie / localStorage) |
| CLI | `~/.config/tars/credentials` (local) or `--token` flag |

Tokens are always JWT (self-signed or signed by an OIDC issuer):

```json
{
  "iss": "https://sso.example.com",
  "sub": "user@example.com",
  "tenant_id": "acme-corp",
  "scopes": ["tenant:acme:read", "tools:invoke:fs.*"],
  "exp": 1746180000
}
```

The server verifies them uniformly via `tars-security`.

## 13. Error Mapping

Each language expresses the same logical errors using its native idiom:

| Logical error | Rust | Python | TypeScript | gRPC status | HTTP |
|---|---|---|---|---|---|
| Authorization failed | `Err(IamDenied)` | `IamDeniedError` | `IamDeniedError` | `PERMISSION_DENIED (7)` | 403 |
| Resource missing | `Err(NotFound)` | `NotFoundError` | `NotFoundError` | `NOT_FOUND (5)` | 404 |
| Budget exhausted | `Err(BudgetExceeded)` | `BudgetExceededError` | `BudgetExceededError` | `RESOURCE_EXHAUSTED (8)` | 429 |
| Cancelled | `Err(Cancelled)` | `CancelledError` | `CancelledError` | `CANCELLED (1)` | 499 |
| Provider error | `Err(ProviderError)` | `ProviderError` | `ProviderError` | `INTERNAL (13)` | 502 |
| Internal error | `Err(Internal)` | `RuntimeError` | `Error` | `INTERNAL (13)` | 500 |

Every error object carries:
- `code`: string enum value (machine-readable)
- `message`: human-readable
- `trace_id`: for SRE correlation
- `details`: structured supplementary info

## 14. Streaming Protocols

How the four protocols handle streaming:

| Protocol | Streaming mechanism | Usage constraint |
|---|---|---|
| Rust | `BoxStream<TrajectoryEvent>` | Caller holds the stream; Drop triggers cancel |
| HTTP | SSE (`Content-Type: text/event-stream`) | EventSource handles reconnect automatically / heartbeat required |
| gRPC | server streaming `stream TrajectoryEvent` | Client cancel via stream close |
| Python | `async for event in runtime.subscribe(id)` | asyncio compatible |
| TypeScript | `for await (event of stream)` | Standard AsyncIterable |

Cross-protocol guarantees:
- **Same event schema for the same stream** (OpenAPI / proto / pyi / d.ts kept in sync)
- **Terminal event identification**: `completed` / `failed` / `cancelled` are always the last event in the stream, recognizable across all protocols
- **Cancel semantics**: caller closes stream → server receives cancel signal → Doc 02 §5 cancel chain triggered

## 17. Anti-Pattern Checklist

1. **Don't let any single binding offer "extra" features** — every feature comes from the Rust core; bindings are projections.
2. **Don't write business logic in Python/TS bindings** — logic lives in Rust; bindings only translate.
3. **Don't assume every client will upgrade** — keep deprecated APIs alive for at least 12 months.
4. **Don't expose internal IDs other than trace_id in the HTTP API** — use stable external IDs.
5. **Don't pretend WASM can run the full Runtime** — clearly document it as client-only.
6. **Don't share memory across PyO3 / napi-rs bindings** — copy data on each FFI call; simple but safe.
7. **Don't guard FFI entry points with a global lock** — use Arc + fine-grained internal locks.
8. **Don't let HTTP clients poll task status** — must use SSE / WebSocket streaming.
9. **Don't hard-code base_url in the SDK** — let the user configure it.
10. **Don't ignore SDK version negotiation** — detect server version at startup.
11. **Don't maintain schemas independently across bindings** — must be generated from the Rust single source of truth.
12. **Don't "exempt" any binding from a conformance test** — all must pass before release.
13. **Don't change CLI output format in a patch release** — human-readable output is also API; change with care.
14. **Don't skip heartbeat on SSE** — without events, proxies kill the connection silently and the client is unaware.
15. **Don't translate only the message of cross-language errors** — must include a structured code so programs can recognize them.

## 18. Contracts with Up- and Downstream

### Upstream (Client / SDK user) Commitments

- Handle all error codes; do not assume "this won't happen"
- SSE clients handle reconnect correctly
- Respect rate limits; back off on 429
- Upgrade SDK to at least the same major version as the server

### Downstream (Rust core trait) Commitments

- All public APIs must be `Send + Sync` and usable across threads
- Streaming event types must be serializable (serde / proto / pyo3)
- Error types implement `Display + Error + Send + Sync + 'static`
- Do not expose internal implementation details like `Arc<Mutex<...>>`

### Binding Maintenance Contract

- Any change to the Rust core public API must update every binding in lockstep
- New APIs land in Rust + HTTP first; Python/TS may follow 1-2 minor releases later
- The binding maintainer has final decision authority on that binding (idiomatic translations may differ)
