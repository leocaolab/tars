# 文档 33 — 为什么 TARS 不做 MCP

状态：**已决策 —— 2026-07 定论。** TARS **不提供任何 MCP**，client 或 server 都没有，而且这是一个我们引以为傲的选择，不是一个让我们羞于启齿的 TODO。这份 ADR 就是要讲清楚为什么 —— 好让未来任何一位贡献者，在某个被时髦风气冲昏头的软弱瞬间，都不会把它硬塞进内核"好让项目看起来更完整"。

> 一句话版本：MCP 是一个已经失败过三次的想法的第四次转世。它没有消除复杂度，只是把复杂度挪了个地方。作为一种*类型*，它是一袋扁平的 `Json → Text`，没有任何办法把两样东西组合出第三样。作为一块*基础设施*，它是迟到三十年、而且做得更差的 HTTP、REST 和 RPC —— 甚至算不上一个合格的 microservice：不安全、不可扩展、没经过实战检验。它唯一货真价实的优点，是有那么几个封闭的 app 已经知道怎么拨通它。那是一个分发事实，不是一个工程事实。TARS 保留自己那套有类型、可组合的脊梁，拒绝让这东西靠近它。

---

## 1. 万能插头的简史，以及它身后那片墓地

大约每隔十年,软件界最优秀的头脑就会聚到一起去解决同一个问题 —— *我怎么让我的系统跟你的系统对话?* —— 而每隔十年,他们都会伸手去够同一个答案:一层万能的 middleware,它要把整个世界裹进一个协议里,从此万物便会简单地、魔法般地、开口说话。

上世纪 90 年代,那是 **CORBA**。一门接口定义语言、一堆 object request broker、一个叫 IIOP 的线上协议,再加上那个辉煌的承诺:任何 object 都能调用任何其他 object,任何语言、任何机器,仿佛它们就坐在同一个房间里。它本该改变一切。

接着来了 **SOAP** 和 **WSDL** —— 一个重到需要自带邮差的信封,和一门啰嗦到"大声念一遍都算一种刑罚"的 schema 语言。

再接着是 **Enterprise Service Bus** 和整场 **SOA** 淘金热:整整十年、一整片海洋的咨询费,砸进去把每一个吱吱作响的企业系统都包成一个"service",好让那条 Bus —— 像个坐在正中央的土邦君主 —— 在它们之间路由消息。

它们如今在哪儿?CORBA 是一个用来吓唬应届毕业生的鬼故事。SOAP 是一句笑话。ESB 是 2009 年某份迁移预算里、没人愿意再提起的一行条目。三者死于一模一样的病,而现在正是给这病命名的时候,因为我们即将眼睁睁看着它放倒第四个病人:

1. **万能封装从未消除复杂度 —— 它只是挪了个地方。** 从调用方那儿挪走了,调用方现在轻松了;挪到了某个精疲力竭的可怜人身上,那人的全部工作就是维护这层封装。
2. **一个更简单的底座冒了出来,把它们吃掉了。** 朴素的 HTTP、JSON、一个文本编辑器和一条 `curl` 命令,不带任何繁文缛节就干掉了九成的活儿,那位土邦君主则被悄悄地从后门请了出去。

记住这个念头。它差不多就是整篇评审的全部。

## 2. MCP 登场 —— 顺便看看 agent 现在到底怎么干活

Model Context Protocol 是 2024 年那次万能插头的转世。纸面上它很整洁:一个 server 广告出一张工具清单,一个 client 按名字调用其中一个,JSON 进去,"content" 出来。这个生态确实庞大 —— 公开目录里列着数以万计的 server(去重之后数字要小得多),SDK 每月下载量以千万计。用它布道者不厌其烦的话说,它已经成了"AI 的 USB-C 接口"。

所以在我们把它按到工作台上之前,先来看看一个*有能力的* agent —— 我们真正为之构建的那种东西 —— 在你交给它一个真实任务时会干什么。

它要一个 shell。它敲下 `gh pr view 412 --json files`。它写 `SELECT id, total FROM orders WHERE status = 'stuck'`。它拿着从进程环境里读到的 key 去 `curl` Stripe API。它读 OpenAPI spec,或者读 SDK 的 docstring,然后它就……直接把那东西调了。

它*不会*做的,是向你要一份预先嚼碎的、400 条工具描述的目录,好让它猜该调哪一个。前沿的编码 agent 早就完全绕开了 MCP,因为一个 shell 加一个好用的 SDK,是比一袋 JSON-RPC 封装更强、更快、更诚实的接口。把这幅画面记在脑子里 —— 那个乐呵呵敲着 `gh` 的 agent —— 因为接下来的一切,不过是在拆解:为什么这幅画面赢了。

## 3. 那到底是谁靠它成功了?(没有一个干净的成功案例)

如果 MCP 真的顶用,你会预期看到一个旗舰级胜仗 —— 一次部署,其中*协议本身*就是它奏效的原因。没有这样的例子。只有两种"成功",而两种都不是 MCP 的胜利:

- **Block 的 Goose** 把内部系统 —— DataHub 元数据、一个 service registry、一个 incident 工具 —— 包成 MCP server,让工程师通过同一扇自然语言的门去够到它们全部。挺好。但这里的胜仗是*"给那些从来没有过一个像样 API 的内部系统开了一扇统一的门"*,而一个内部的 gRPC gateway 恰恰能提供这一点。协议在这里是可有可无的。
- **Cloudflare 的 MCP Demo Day** 列队展示了 Asana、Atlassian、Linear、PayPal、Sentry、Stripe 和 Webflow 纷纷发布远程 MCP server。这里的胜仗是*"现在能从 Claude 内部够到它们了"*。那是防御性分发 —— 把你的产品摆到一群被圈住的观众面前 —— 不是投资回报。

每一个"成功"都坍缩成*统一入口*或*从一个封闭宿主里可达*。两者都不是 MCP 这个抽象本身的性质。这就是破绽所在。没有任何人、任何地方,是因为这个抽象好才采纳它的。

---

## 4. 论点

**MCP 是一个 anti-pattern。** 不是"不成熟",不是"尚早"。是 anti-pattern:一个看起来像解法、实际上是"戴着工牌的问题本身"的形状。本文档余下的部分,就是控方的举证,分三部分 —— 它作为一种类型是什么,它作为一块基础设施是什么,以及为什么 TARS 尤其是地球上最不需要它的那个系统。

---

## 5. 类型层面的指控(为什么它在纸面上就丑)

把 MCP 当作*一种类型*认真对待,它就在你手里瘪掉了:

```
Server  ≈  { tools : List<(Name, JsonSchema_in, Description)> }
call    :  (Name, Json) -> Result<Content[], Error>
Content =  Text | Image | Audio | ResourceLink | EmbeddedResource
```

每个工具的有效类型都是 **`Json → Text`**。schema 是运行期元数据,不是静态类型。返回值被 `Text` 主导。`structuredContent` 是一个你得主动选择加入的、事后硬拴上去的补丁。三个后果:

**5.1 —— 它是惰性的。** MCP 贡献了*零*智能。它全部的操作语义就是"一个模型读一句英文,然后*猜*该调用哪个名字"。这个抽象没有属于自己的任何行为;它只有相对于一个外部的、随机的解释器才被定义。把 LLM 拔掉,剩下的就只是一本电话簿。

**5.2 —— 它是 stringly-typed 的。** 一个动词 `call`,作用在一个扁平的命名空间上。没有 coproduct(你没法表达"A *或* B"),没有 product(你没法表达"A 的输出*就是* B 的输入")。每一个领域类型都被抹成 JSON 和散文,所以你永远拿不到一个你能信任的值 —— 你拿到的是一坨你*祈祷*它能解析出来的 blob。这正好是 TARS 自己那个 `json_decode` 的反面,后者整个契约就是*"给它一个 `T`,拿回一个合法的 `T` 或一个有类型的 error"*。MCP 递给你一个名字,然后返回一段你必须重新解析的散文,永远如此。

**5.3 —— "可扩展"意味着更宽,不是更深。** MCP 那被人称颂的可扩展性,就是"往清单里再追加一条 `Json → Text` 条目"。没有 combinator —— 没有 `compose : Tool a b → Tool b c → Tool a c`。两个 MCP 工具没法被拼接成第三个。组合只发生在模型的脑子里,以命令式的方式,每一轮都重新推导一遍。它不是一门代数;它是一堆被拆掉了 compose 按钮的 morphism。一个好抽象的全部要点,就在于它的零件能咔哒一声拼到一起。MCP 把那个扔了,只留下了包装。

**公平起见(steelman):** MCP 不止 `tools/call` —— 还有 `resources`、`prompts`,以及 `sampling` 和 `elicitation` 里那么一丝双向性的味道。但这些在实际生态里几乎没被实现,而且它们只是拓宽了表面积,并没有添上一条 compose 律。一个更大的袋子,终究还是个袋子。

## 6. 那当 client 用总行吧?(Stripe 测试)

"行吧,"拥护者说,"别造 server —— 只*消费*别人的就好。"不行。只要你会写代码,一次直接调用就在每一个要紧的维度上碾压一次 MCP 调用:有类型、在进程内、可 sandbox、没有额外一跳、没有 daemon、没有 schema 往返。那些"高价值"的 server —— GitHub、Playwright、一个数据库 —— 都是对那些本就可以直接、且极其好用的工具的封装:`gh`、Playwright 库、一个 driver。

跑一遍 **Stripe 测试**。*"为什么不直接调 Stripe 的公开 REST API 呢 —— agent 干不了这个吗?"* 它当然干得了。一个有能力的 agent 会读 OpenAPI,直接调 REST。一个 Stripe MCP server 只多添两样东西:**curation**(把几百个 endpoint 修剪成几个带着 LLM 友好文案的"安全"端点 —— 那是护栏,不是能力)和 **替你握着 key**(把它挡在模型的 context 之外)。而这两样,你在自己代码里都做得*更好*。Stripe 其实是 MCP *最糟*的案例,恰恰因为它的 REST API 是世界一流的 —— 已经没有什么留给封装去添了。

于是我们得到了那条统御一切的定律,而它是残忍的:

> **一个 MCP server 的价值,与调用它的那个 agent 的能力成反比。**

agent 越聪明,一份预先嚼碎的目录就越不值钱。MCP 是给一个*弱* agent 搭的脚手架,那种还不会读文档、开 shell、调 API 的 agent。随着 agent 变强 —— 而它们正在以一种吓人的速度变强 —— 脚手架就成了死重。它在结构上,就是一场"押注反对所有人正拼命去建造的那个东西"的赌局。

## 7. 实务层面的论证 —— 以及一段关于证据的诚实声明

上面说的全部是*架构*层面的论证,一个纯粹主义者大可挥挥手把它撇开。所以值得问一句:MCP 在真实世界里到底表现如何。但先来一段刻意的省略,因为这份 ADR 必须是我们能站得住脚去背书的东西:

> **一段关于证据的声明,因为这份 ADR 必须是我们能站得住脚的那种。** 凡是我们引用之处,我们引用的都是我们对着一个*一手*来源核实过的东西:**OWASP MCP Top 10**(一个真实存在的 OWASP Foundation 项目)和具体的 **NVD** CVE 条目。而我们刻意*不*去倚靠的,是那些填满 2026 年博客文章的松散百分比("X% 的 server 存在 Y 漏洞"):它们只能追溯到一些厂商综述,而其底层的调查我们打不开,并且至少有一个被广泛转述的"统计数字",一经核查,原来是一次模型外推,随后在低权威度的网络上来回弹跳,直到看起来像个事实。所以我们锚定在那个具名的框架和那条 CVE 上,并且把这些百分比标注为厂商自报,而不是当作事实断言。让人松一口气的是,这个论证本来就几乎不需要数字 —— 那些失败模式是*结构性*的,而 OWASP 早已把它们编成了目录。

**7.1 —— 那所谓的采用,是一场人质事件,不是一场胜利。** MCP 确实无处不在 —— 各家企业竖起 gateway 去把 Snowflake、Salesforce、内部 wiki 暴露出来。但*问问他们为什么*。没有一家是因为这个抽象优雅才这么干的。他们这么干,是因为他们的员工生活在**封闭宿主**里 —— Cursor、Claude Desktop、Microsoft Copilot —— 而那些宿主只认这一种插头。没有 MCP endpoint,就够不到你自己的数据。那不是一个协议凭本事赢下的;那是一座收费站。"AI 的 USB-C"只是一种更好听的说法,它真正的意思是*"这台电器唯一接受的插座"*。

**7.2 —— 安全上的失败模式是结构性的 —— 而 OWASP 早已把它编好了目录。** 你不必听我一面之词去信这些失败类别:OWASP 如今发布了一份专门的 **MCP Top 10**(MCP01–MCP10:2025,一个真实的 OWASP Foundation 项目)。读一读这张清单 —— Token 与 Secret 泄露、Agent 权限过度、Tool Poisoning、供应链 / 依赖篡改、Command Injection、Prompt Injection、鉴权不足、缺乏 Audit、Shadow MCP Server、Context 过度共享 —— 它读起来就像本文档的一份摘要。每一条都是这个设计*可预见*的后果:一个无类型的 server,把一个随机模型的输出当作控制输入接进来,并作为数以千计、按名字安装的社区 package 发布出去。path traversal 和 command injection,是当"模型要了一个文件 / 一条命令,而封装照办了"时你会得到的东西;供应链 typosquatting,是当"安装那个你只记得一半名字的 `some-mcp-server`"时你会得到的东西,而现在还附带着预先授权的企业凭证;通过工具*描述*进行的 prompt injection,是这个协议故意开出来的一扇正门。而且这不是假想 —— **CVE-2025-6514**(NVD;CVSS **9.6**;由 JFrog 发现)是一个有据可查的、在 *client* 机器上仅仅因为连接到一个不可信 MCP server 就发生的 OS 命令注入 RCE。厂商的分析(例如 Cycode)给这一切附上了骇人的流行度百分比;我们把那些标注为厂商自报,而不去倚靠它们,因为一份具名的 OWASP Top 10 加上一个 CVSS-9.6 的 client RCE,不需要一个有争议的数字就已经说明了问题。这一切都不是运气不好;这是它的形状。这也正是 TARS 对每一个 delegate 都上 write-jail 的全部理由 —— 那些转而信任了这袋工具的团队,正是眼下在向董事会解释一起事故的那些人。

**7.3 —— 它甚至算不上一个合格的 microservice —— 它是被拙劣重造的 HTTP 和 RPC。** 剥掉品牌,MCP 就是 JSON-RPC over HTTP(或 stdio),再在上头硬拴一个 `tools/list` 发现调用。也就是说:它是在**重新发明 HTTP、REST 和 RPC**,迟到三十年,而且离追上那些东西闭着眼睛都能做的事还有很长的路要走。而且它重造得*很差*:一旦它伸手去够*有状态*的 session,它就变成了它所克隆之物的一份不称职的拷贝。任何把每个 session 的状态钉在 server 内存里的协议,都在跟一个普通的 load balancer 对着干 —— 横向扩展需要 sticky routing,一次断开的连接会带走那个 session,连同模型的 context 一起搭进去。REST 在上世纪 90 年代就通过*无状态*恰好解决了这个问题;MCP 把它*解回去了*。这不是什么奇异的边缘情况;它是任何人把一个有状态协议当真正的 service 来跑时撞上的第一堵墙 —— 一个永恒的性质,不需要 2026 年的引用。一个跟 load balancer 对着干的 middleware 不是 middleware。它是一段爆红的 demo。

**7.4 —— 它是复杂度守恒,打扮成了创新。** 这是承重的一点。MCP 没有删掉任何复杂度 —— 它是在*铲*复杂度,从 agent 的桌上铲下来,铲到那个从此得永远维护 `Stripe-MCP-Server`、`DB-MCP-Server`、`K8s-MCP-Server` 的人身上,追着每一次上游 API 变更跑。也就是说:它是**被掘尸出土的 ESB 和 SOA gateway**。世界上有数以万计的 API;计划是把它们全部手工包起来,并且保持更新?当然不是 —— 而这恰恰是为什么这个协议自己的厂商只积极维护屈指可数的几个参考 server,而"生态系统"的其余部分是一片周末玩具的垃圾填埋场。没人想当那个维护封装的可怜灵魂。他们从来都不想。这就是为什么 CORBA 死了。

## 8. 为什么 TARS 尤其连这个插头都不需要

现在到了闭环的部分。MCP 之所以存在,是因为 Claude Desktop 和 ChatGPT 是**漂浮在你城墙之外的黑盒**—— 它们物理上碰不到你的数据库,所以你被迫在自己的网络里竖起一个 server,给它们凿一个洞让它们伸手进来够。

**TARS 不在墙外。TARS 嵌在你自己的 service 或 binary 里面。** 而这句话你一旦大声说出来,MCP 存在的全部理由就蒸发了:

- **凭证?** TARS 跑在你后端*内部* —— 就在那台已经握着 IAM role 的机器上,紧挨着那个已经在进程环境里的 secret。模型发出意图;底下 TARS 的 Rust 去执行调用。凭证从不进入 LLM context,因为它压根不需要离开进程。
- **连接池?** TARS *就是*你 service 的一部分。你的 Redis pool、你的 Postgres pool —— 已经在那儿了。`Pipeline` 直接复用宿主的 pool。没有什么留给一个独立的 MCP server 去"管理"。
- **Audit?** TARS 自带一套 MELT(metrics、events、logs、traces)和一套有类型的 error 层级,而且因为它是**在进程内**执行的,它能以近乎零的成本记录下真实的调用栈 —— 比在线缆上 tcpdump JSON-RPC 清晰一千倍。

为了安全地够到外部世界而把 TARS 裹进 MCP,就像在一座堡垒内部支起一顶塑料帐篷。TARS *一开始*就在墙后面。

---

## 9. 裁决

- **作为核心架构:** MCP 是一个漏筛子,祖上是一串死掉的 middleware,案底是一摞活着的 CVE。把它挡在核心之外不是谨慎,是卫生。**驳回。**
- **作为一个外层插头:** *假如*有一天我们真心想服务那些被困在 Cursor 或 Claude Desktop 里的用户,我们*可以* —— 不情不愿地 —— 在最外层边界上拴一个一次性的 MCP *server* 适配器:把一个真实的 `T` 坍缩成 MCP 的 `Json → Text`,递出去,然后把这次坍缩丢掉。它几乎不花什么成本(它不过是 JSON-RPC),而它相较于朴素 gRPC 也几乎挣不到什么,除了"那个封闭宿主已经知道拨这个号"。它是一个 gateway 转换器,不是一个架构原语,而且它永远不会被当作一个 feature 来提起。
- **互操作性上的诚实:** TARS 今天既不支持 MCP 也不支持 A2A,而这没问题。我们不会为了看起来完整而回填兼容性徽章。缺席就是真相;我们如实说出来。

## 10. 什么能让我们改变主意

因为一份诚实的 ADR 会点出它自己的证伪条件。我们只会在下列情况下重新审视那个 *server 适配器*(绝不是 client):

1. 一个我们在乎的封闭宿主,成为了 TARS 的 sandboxed 执行的一条**实质性分发渠道**,而 MCP 是唯一的入口。这是一个分发触发条件,不是技术触发条件。
2. MCP 长出了一个**真正的组合原语** —— 有类型、带保证的 tool-to-tool 连接 —— 并且不再是一袋扁平的 `Json → Text`。那时 §5.3 就不再成立,这东西就值得凭它自己的本事重新读一遍了。

在这两条中的任何一条为真之前,答案都是一个愉快的、证据充分的**不**。

**证据,以及它是怎么被核实的。** 这份 ADR 所倚靠的两条外部主张,都对着一手来源核实过:
[OWASP MCP Top 10](https://owasp.org/www-project-mcp-top-10/)
([项目仓库](https://github.com/OWASP/www-project-mcp-top-10))是一个真实的 OWASP
Foundation 项目,而
[CVE-2025-6514](https://nvd.nist.gov/vuln/detail/CVE-2025-6514) 是一条真实的 NVD 条目
(CVSS 9.6,package `mcp-remote`,OS 命令注入,由 JFrog 报告)。我们刻意*不*去倚靠那些在 2026 年博客文章里流传的松散漏洞百分比("X% 的 server 存在 Y 漏洞"):它们只能追溯到一些厂商综述,而其底层的调查我们打不开,并且至少有一个被广泛转述的数字,原来是一次模型外推,在低权威度的网络上来回弹跳,直到看起来像个事实。如果日后有一份可信的第一方审计落地,请在这里引用它,连同一手链接,而不是一次博客转述。
