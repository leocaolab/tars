│                                                                              │

│ 存储，Key 是 `event_id`，Field 是 `ticket_id`，Value 是                      │
│ `user_id`（预扣成功后记录）：                                                │
│                                                                              │
│ *   **Key:** `tickets:event:1001` (约 20 bytes)                              │
│ *   **Field (Ticket ID):** `100000001` (长整型或字符串，约 10 bytes)         │
│ *   **Value (User ID/Status):** `500000001` (已售出的用户 ID，约 10 bytes)   │
│ *   **Redis 额外开销：** 每个 Hash                                           │

│                                                                              │

│                                                                              │

│ ### 2. 自适应调节反馈算法：梯度降级算法 (Gradient Limit)                     │
│                                                                              │
│ 我会参考 **Netflix 的 ConcurrencyLimit** 逻辑，实现一个基于 **Little's Law   │
│ (利特尔法则)** 的自适应调节器。                                              │
│                                                                              │
│ #### 核心逻辑描述：                                                          │
│ 咱们不设定一个死的人数上限（比如每秒 1000 人），而是动态计算一个**“窗口值    │
│ (Limit)”**。                                                                 │
│                                                                              │
│ **算法步骤：**                                                               │
│ 1.  **采样：** 每隔一个周期（比如 1s），统计当前的                           │
│ $RT_{curr}$（平均响应时间）和 $RT_{min}$（历史最优响应时间）。               │
│ 2.  **计算梯度 (Gradient)：**                                                │
│     $$Gradient = \frac{RT_{min}}{RT_{curr}}$$                                │
│     *   如果 $RT_{curr}$ 升高，$Gradient$ 就会小于 1，表示系统开始拥堵。     │
│ 3.  **更新窗口值 $L$ (New Limit)：**                                         │
│     $$L_{next} = L_{curr} \times Gradient + \text{QueueSize}$$               │
│     *   这里的 `QueueSize` 是一个小常数，用来给系统留一点“探索”空间。        │
│ 4.  **平滑处理：** 为了防止抖动，我会给这个 $L_{next}$ 加一个平滑系数（如    │
│ 0.9），避免排队机开关猛开猛合。                                              │
│                                                                              │
│ #### 具体的伪代码实现：                                                      │
│ ```python                                                                    │
│ # 每秒触发一次                                                               │
│ def adaptive_throttle(current_rt, min_rt, current_limit):                    │
│     # 1. 计算背压梯度                                                        │
│     gradient = min_rt / max(min_rt, current_rt)                              │
│                                                                              │
│     # 2. 结合梯度调整限流窗口                                                │
│     # 如果 RT 变大，gradient < 1，new_limit 会收缩                           │
│     new_limit = current_limit * gradient + log10(current_limit)              │
│                                                                              │
│     # 3. 边界保护：不低于最小并发数，不超过系统硬上限                        │
│     new_limit = clamp(new_limit, MIN_LIMIT, MAX_LIMIT)                       │
│                                                                              │
│     return new_limit                                                         │
│ ```                                                                          │
│                                                                              │
│ ---                                                                          │
│                                                                              │
│ ### 3. 权衡与方案对比 (Trade-offs)                                           │
│                                                                              │
│ *   **为什么不用简单的 AIMD (和 TCP 一样加增乘减)？**                        │
│     *   **AIMD**                                                             │
│ 的问题在于它太“激进”了，一旦超时就减半，这会导致排队机的吞吐量剧烈抖动，用户 │
│ 体验非常差（一会儿进得去，一会儿死活进不去）。                               │
│     *   **梯度算法 (Gradient)**                                              │
│ 更加平滑，它是根据响应时间的退化程度来等比例收缩，能更好地维持在一个“临界饱  │
│ 和状态”。                                                                    │
│                                                                              │
│ *   **这个方案的缺陷：**                                                     │
│     如果 DB 出现了那种“伪死机”（连接不报错，但执行极慢），$RT_{min}$         │
│ 的基准可能会失效。这时候我可能需要引入一个**“强制熔断机制”**作为兜底，如果   │
│ $T_{wait}$ 超过一个硬阈值（比如 2 秒），直接切断排队机的放行，快速失败。     │
